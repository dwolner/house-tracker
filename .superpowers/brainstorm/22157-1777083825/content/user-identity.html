How should users identify themselves?

Each approach handles scoring preferences + notification subscriptions differently. Pick the one that feels right.

A

localStorage + email for notifications

Scoring weights live in the browser. User enters email once to subscribe to notifications for their locale. Email saved to localStorage so they don't re-enter it.

Pros

  • Zero new DB complexity for weights
  • Instant — no "profile" concept at all
  • Only adds a subscribers table (email + locale)

Cons

  • Scoring weights lost if browser data cleared
  • Can't move weights between devices
Flow: Open app → weights are default → adjust sliders → auto-saved to localStorage. Enter email in notification panel → server stores {email, locale_id}.
B

Email-keyed profiles (server-side)

User enters email → creates/loads a profile. Scoring weights stored in DB keyed by email. On return visit, email auto-filled from localStorage, weights loaded from server.

Pros

  • Weights persist across devices & browsers
  • Email = natural identity (no passwords)
  • Notifications + weights unified under one record

Cons

  • New users table + API endpoints
  • User must enter email to get personalized scores
  • Slightly more backend work
Flow: Open app → "Enter your email to load preferences" → weights load from server. Adjust sliders → auto-save to server. Notifications go to that email automatically.
C

URL token profiles

Server generates a unique token per user. App URL becomes /?u=abc123. Bookmark it and your preferences follow. Email stored in profile for notifications.

Pros

  • No "login" at all — URL is identity
  • Easy to share with friend: "here's your link"
  • Works across devices via the URL

Cons

  • Lose the URL = lose your profile (recoverable by email)
  • Feels a bit awkward in practice
  • Similar backend complexity to Option B
Flow: First visit → server creates profile + redirects to /?u=abc123. You bookmark it. Enter email for notifications. Friend gets their own URL.
My recommendation: B — email-keyed profiles. Email is already how notifications work, so it's the natural identity. The extra backend work is minimal (one table, two endpoints), and weights following the user across devices is worth it for a shared app.
